# Frequently Asked Questions

Common questions about Isar Plus and Flutter databases.

## General Questions

<Accordions>
  <Accordion title="Why do I need a database?">
    Even with a backend database, local storage is essential for:
    
    - **Offline functionality** - Work without internet (subway, plane, rural areas)
    - **Performance** - Instant data access without network latency
    - **User experience** - Smooth app operation regardless of connection
    - **Data caching** - Reduce server load and bandwidth usage
    - **Privacy** - Keep sensitive data on device
    
    <Callout type="success">
      Users expect apps to work seamlessly even with poor or no connection!
    </Callout>
  </Accordion>

  <Accordion title="Isar Plus vs Isar?">
    Isar Plus is a community-maintained fork of the original Isar database:
    
    **Isar Plus Advantages:**
    - ‚úÖ Active maintenance and updates
    - ‚úÖ SQLite backend for better web support
    - ‚úÖ Community-driven development
    - ‚úÖ Regular bug fixes and improvements
    - ‚úÖ Enhanced web performance with OPFS
    
    **Migration:**
    Most code is compatible. See the [Migration Guide](/docs/recipes/migrate_from_isar3).
    
    <Callout>
      We recommend Isar Plus for all new projects and migrations from Isar 3.
    </Callout>
  </Accordion>

  <Accordion title="Isar Plus vs Hive?">
    Isar Plus is significantly more powerful:
    
    | Feature | Isar Plus | Hive |
    | --- | --- | --- |
    | Queries | ‚úÖ Advanced | ‚ùå Limited |
    | Indexes | ‚úÖ Multi-type | ‚ùå Basic |
    | Relationships | ‚úÖ Yes | ‚ùå No |
    | Transactions | ‚úÖ ACID | ‚ö†Ô∏è Basic |
    | Type Safety | ‚úÖ Full | ‚ö†Ô∏è Partial |
    | Performance | ‚ö° Excellent | ‚úÖ Good |
    | Web Support | ‚úÖ Full | ‚úÖ Full |
    
    <Callout type="success">
      Isar Plus was originally created as a Hive replacement and is now production-ready!
    </Callout>
  </Accordion>

  <Accordion title="Isar Plus vs SQLite?">
    Both are excellent choices:
    
    **Isar Plus:**
    - üöÄ Easier to use (no SQL)
    - üíÉ Type-safe queries
    - ‚ö° Often faster for complex queries
    - üéØ Built for Dart/Flutter
    - üîó Native relationships
    
    **SQLite:**
    - üìö More established
    - üõ†Ô∏è More tools available

    <Callout>

  </Accordion>

## Performance Questions

<Accordions>
  <Accordion title="Is Isar Plus fast enough?">
    Yes! Isar Plus is among the fastest mobile databases:
    
    **Benchmarks:**
    - Insert: 50,000+ records/sec
    - Query: Sub-millisecond for indexed queries
    - Update: Similar to insert performance
    
    **Optimization Tips:**
    - Use indexes for frequently queried fields
    - Batch operations in single transactions
    - Use lazy loading for large queries
    
    <Callout type="success">
      Isar Plus is fast enough for 99% of use cases. If you have issues, you're probably doing something wrong!
    </Callout>
  </Accordion>

  <Accordion title="How do I optimize performance?">
    Follow these best practices:
    
    ```dart
    // ‚úÖ Use indexes
    @collection
    class User {
      Id? id;
      @Index() late String email;
    }
    
    // ‚úÖ Batch operations
    await isar.writeTxn(() => isar.users.putAll(users));
    
    // ‚úÖ Use where clauses
    await isar.users.where().emailEqualTo('x').findAll();
    
    // ‚ùå Avoid
    for (var user in users) {
      await isar.writeTxn(() => isar.users.put(user));
    }
    ```
    
    See [Indexes](/docs/indexes) and [Transactions](/docs/transactions) for details.
  </Accordion>

  <Accordion title="Why are where clauses better than filters?">
    Where clauses use indexes for direct lookups:
    
    ```dart
    // ‚ö° Fast - uses index (O(log n))
    await isar.users.where()
      .emailEqualTo('john@example.com')
      .findAll();
    
    // üêå Slow - scans all records (O(n))
    await isar.users.filter()
      .emailEqualTo('john@example.com')
      .findAll();
    ```
    
    **When to use each:**
    - Where: Indexed fields, large datasets
    - Filter: Non-indexed fields, complex conditions
    
    <Callout type="info">
      You choose which index to use. You know your data best!
    </Callout>
  </Accordion>
</Accordions>

## Usage Questions

<Accordions>
  <Accordion title="Do I need to use indexes?">
    No, but they're highly recommended:
    
    **Without indexes:**
    - ‚úÖ Simpler schema
    - ‚úÖ Faster writes
    - ‚ùå Slower queries
    - ‚ùå No where clauses
    
    **With indexes:**
    - ‚ö° Much faster queries
    - ‚úÖ Where clause support
    - ‚úÖ Efficient sorting
    - ‚ö†Ô∏è Slightly slower writes
    
    <Callout>
      For small collections (&lt;1000 items), indexes might not be necessary. For larger collections, they're essential!
    </Callout>
  </Accordion>

  <Accordion title="Can I use Isar Plus with multiple isolates?">
    Yes! Isar Plus has excellent multi-isolate support:
    
    ```dart
    // Main isolate
    final isar = await Isar.open([UserSchema]);
    
    // Background isolate
    await compute((schemas) async {
      final isar = await Isar.open([UserSchema]);
      // Use same instance
      await isar.users.where().findAll();
    }, [UserSchema]);
    ```
    
    See the [Multi-Isolate Recipe](/docs/recipes/multi_isolate) for details.
  </Accordion>

  <Accordion title="How do I handle migrations?">
    Isar Plus handles most migrations automatically:
    
    **Automatic:**
    - Adding new fields
    - Removing fields
    - Adding/removing indexes
    - Changing field types (with data loss)
    
    **Manual handling needed:**
    - Complex data transformations
    - Preserving data when changing types
    
    ```dart
    // Data migrations
    final users = await isar.users.where().findAll();
    await isar.writeTxn(() async {
      for (var user in users) {
        // Transform data
        user.newField = transformOldField(user.oldField);
        await isar.users.put(user);
      }
    });
    ```
    
    See [Data Migration Recipe](/docs/recipes/data_migration).
  </Accordion>

  <Accordion title="Can I use Isar Plus on Web?">
    Yes! Isar Plus has full web support:
    
    **Storage:**
    - Chrome/Edge 102+: OPFS (fast)
    - Safari/Firefox: IndexedDB (compatible)
    
    **Limitations:**
    - Must use async APIs
    - Some text helpers unavailable
    
    ```dart
    // ‚úÖ Works on all platforms
    await isar.writeTxn(() => isar.users.put(user));
    
    // ‚ùå Doesn't work on web
    isar.writeTxnSync(() => isar.users.putSync(user));
    ```
    
    See [Limitations](/docs/limitations) for details.
  </Accordion>

  <Accordion title="How do I debug Isar issues?">
    Several debugging tools available:
    
    **Isar Inspector:**
    ```dart
    final isar = await Isar.open(
      schemas: [UserSchema],
      inspector: true, // Enable in debug mode
    );
    ```
    
    **Logging:**
    ```dart
    // Log all operations
    Isar.logger = (level, message) {
      print('[$level] $message');
    };
    ```
    
    **Manual inspection:**
    ```dart
    // Count records
    print('Users: ${await isar.users.count()}');
    
    // Check data
    final users = await isar.users.where().findAll();
    print('All users: $users');
    ```
  </Accordion>
</Accordions>

## Size & Storage Questions

<Accordions>
  <Accordion title="Does Isar Plus increase app size?">
    Yes, but minimally:
    
    **Native apps:**
    - iOS/Android: ~1-1.5 MB
    - Desktop: Similar
    
    **Web:**
    - isar.wasm: ~1 MB (optimized)
    - isar.js: ~50 KB (loader)
    
    <Callout>
      The functionality you get far outweighs the size cost!
    </Callout>
  </Accordion>

  <Accordion title="How much storage does Isar Plus use?">
    Storage depends on your data:
    
    **Factors:**
    - Number of records
    - Size of each record
    - Number of indexes
    - List fields
    
    **Typical sizes:**
    - 1000 small objects: ~100 KB
    - 10,000 medium objects: ~2 MB
    - 100,000 large objects: ~50 MB
    
    ```dart
    // Check database size
    final file = File(isar.path);
    final sizeInBytes = await file.length();
    print('DB size: ${sizeInBytes ~/ 1024} KB');
    ```
  </Accordion>

  <Accordion title="Can I compact the database?">
    Yes, using compact on launch:
    
    ```dart
    final isar = await Isar.open(
      schemas: [UserSchema],
      compactOnLaunch: CompactCondition(
        minFileSize: 100 * 1024 * 1024, // 100 MB
        minRatio: 3.0, // Compact if 3x larger than needed
      ),
    );
    ```
    
    Manual compaction not supported, but SQLite handles this efficiently.
  </Accordion>
</Accordions>

<Accordions>

    See [Quick Start](/docs/quickstart) for setup.
  </Accordion>

  <Accordion title="Transaction errors">
    ```dart
    // ‚ùå Error: Transaction already completed
    await isar.writeTxn(() async {
      await isar.users.put(user);
      throw Exception('Error');
      // Transaction ends here
      await isar.users.put(user2); // Won't execute
    });
    
    // ‚úÖ Fix: Don't reuse failed transaction
    try {
      await isar.writeTxn(() async {
        await isar.users.put(user);
      });
    } catch (e) {
      // Start new transaction
      await isar.writeTxn(() async {
        await isar.users.put(userBackup);
      });
    }
    ```
  </Accordion>

  <Accordion title="'Cannot open database' error">
    Possible causes:
    
    1. **Invalid directory:**
       ```dart
       // Get valid directory
       final dir = await getApplicationDocumentsDirectory();
       final isar = await Isar.open(
         schemas: [UserSchema],
         directory: dir.path,
       );
       ```
    
    2. **Permissions issue:**
       - Check app permissions
       - Verify directory is writable
    
    3. **Corrupted database:**
       ```dart
       // Delete and recreate
       final file = File('path/to/default.isar');
       if (await file.exists()) {
         await file.delete();
       }
       ```
  </Accordion>
</Accordions>
